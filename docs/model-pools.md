## 当前实现方式

- 后端维护两个模型池：`STANDARD_MODELS` 和 `LIGHT_MODELS`。
- 每次点击翻译时，前端生成一次 `modelSessionId`，后端基于它对当前模型池生成本次任务的稳定随机顺序；测试期间前端会显示这份顺序。
- 术语提取固定走标准模型池，因为它对一致性和质量更敏感。
- 普通翻译根据“快速模式”开关选择标准或轻量模型池。
- 韩文残留修正/译文复核复用对应模式的模型池和同一个 `modelSessionId` 随机顺序；复核时会把对应韩文原文、译文 chunk、术语表和备选翻译段落标记传给模型，用于最小范围修正韩文残留和逻辑明显奇怪的术语误译。敏感/机械翻译 fallback 段落会跳过模型复核，仅保留原文辅助人工检查。如果当前模型返回非额度类 400 错误，会继续尝试本次随机顺序里的备用模型，避免因单个模型不支持当前消息格式或参数而直接失败。
- 如果 API 报错被识别为额度/余额/计费相关错误，后端会把当前模型标记为已耗尽，并按本次随机顺序切换到同一模型池里的备用模型重试。
- 如果普通翻译或术语提取在主模型池轮询后仍抛出供应商错误，后端会改用 `SENSITIVE_FALLBACK_MODELS`，并在每次触发时完全随机打乱这个兼容模型池的轮询顺序。
- 如果普通翻译遇到其他非额度错误，会先尝试切小 chunk；仍失败时轮询 `SENSITIVE_FALLBACK_MODELS`，最后的最后才使用 Google Translate fallback。术语提取没有 Google Translate fallback，兼容模型池也失败时会降级为空术语列表。
